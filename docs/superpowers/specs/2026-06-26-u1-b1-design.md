## Part A — U1: Django Admin completion

Existing admin already covers `ModelAdmin` (list_display / list_filter / search_fields), inline, and basic actions for: `User`, `ParentStudent`, `Attempt`, `Tag`, `KompetensiDasar`, `Question`, `Quiz`, `Subject`, `Topic`. Remaining gaps from ui-improvement-plan §5.4:

### A1. Question JSON import (F-08)
- Add an import entry point to `QuestionAdmin` via `get_urls()` + a changelist button (`change_list_template`).
- Upload form: a small Django `Form` with a `FileField` (JSON) and a `Topic` selector (fallback when JSON lacks topic mapping).
- Parsing reuses the logic of the existing `import_questions` management command. Extract that logic into a reusable function `apps/questions/importers.py::import_questions_from_data(data, default_topic=None, created_by=None)` returning a summary (created count, errors). The management command and the admin view both call it.
- On success: `messages.success` with counts; on parse error: `messages.error`, no partial commit (wrap in `transaction.atomic`).

### A2. Permission groups (F-23)
Extend `seed_initial_data` to assign permissions to the three existing groups (idempotent):
- `content_manager`: add/change/view on `questions.Question`, `questions.Tag`, `questions.KompetensiDasar`, `subjects.Subject`, `subjects.Topic`, `quizzes.Quiz`. Plus (after B1) `academic.*` content models.
- `user_manager`: add/change/view on `accounts.User`, `accounts.ParentStudent`, and (after B1) `academic.Enrollment`.
- `finance`: stub — no models until B7 (`subscriptions`). Leave group empty with a comment.

Permission assignment must tolerate missing models (e.g. run before B1 migrated) — look up `Permission` defensively and skip silently if the content type does not exist yet.

## Part B — B1: Academic foundation (expand + migrate)

New app `apps/academic`.

### B1.1 Models
- `EducationLevel`: `code` (SD/SMP, unique), `name`, `order`.
- `Grade`: `level` FK→EducationLevel, `number` (1–9), `label` (e.g. "Kelas 1"), `order`. Unique `(level, number)`.
- `AcademicYear`: `name` (e.g. "2025/2026"), `start_date`, `end_date` (nullable), `is_active` (only one active enforced in `save`/admin). Unique `name`.
- `GradeSubject`: M2M `Grade` × `subjects.Subject` (+ `order`). Unique `(grade, subject)`.
- `Enrollment`: `student` FK→`accounts.User`, `grade` FK→Grade, `academic_year` FK→AcademicYear, `status` (`active`/`completed`), `created_at`. Unique `(student, academic_year)`.

### B1.2 Expand (nullable FK additions)
Add `grade_ref = models.ForeignKey('academic.Grade', null=True, blank=True, on_delete=models.SET_NULL, related_name='+')` to:
- `accounts.User`
- `subjects.Subject`
- `questions.KompetensiDasar`
- `quizzes.Quiz`
- `quizzes.QuizSession`

Add `topic = models.ForeignKey('subjects.Topic', null=True, blank=True, on_delete=models.SET_NULL, related_name='kompetensi_dasar')` to `questions.KompetensiDasar` (KD→Topic per §3.3).

Keep existing `grade:int` columns untouched (contract = B8).

### B1.3 Migrate (data migration)
1. Seed `EducationLevel` (SD, SMP), `Grade` (SD: 1–6, SMP: 7–9), default `AcademicYear` (active). Logic lives in a reusable function callable from both `seed_initial_data` command and the migration (use `apps.get_model` inside migration, not direct import).
2. Backfill `grade_ref` from `grade:int` on User, Subject, KompetensiDasar, Quiz, QuizSession (map int → Grade by `number`).
3. Backfill `Enrollment`: for each `User(role=student)` with a grade, create `Enrollment(student, grade=mapped, academic_year=active, status=active)`.
4. KD `topic`: leave null (no reliable int→topic mapping); admin reassigns later. Documented, not backfilled.

Migration must have `reverse_code` (delete seeded/backfilled rows) — reversible.

### B1.4 Code helper (no UI rewire yet)
Add `User.current_enrollment` property → active Enrollment for active year (or None). UI switch from `user.grade` → Enrollment is **U2**, out of scope here. Reads elsewhere keep using `grade:int`; `grade_ref` populated and available for U2.

### B1.5 Admin
Register `EducationLevel`, `Grade`, `AcademicYear`, `GradeSubject`, `Enrollment` with sensible list_display/filter/search. `GradeSubject` inline on `Grade`. Enforce single active `AcademicYear`.

## Testing (pytest-django)

Setup:
- Add `pytest`, `pytest-django` to `requirements/` (dev/test).
- `pytest.ini` / `conftest.py`: `DJANGO_SETTINGS_MODULE=config.settings.base`, a test DB.

Tests (`apps/academic/tests/`):
- Seed correctness: SD has grades 1–6, SMP 7–9; one active AcademicYear.
- Idempotent seed: running twice creates no duplicates.
- Backfill: a User with `grade=3` gets `grade_ref.number == 3`; same for Subject/KD/Quiz/QuizSession.
- Enrollment backfill: student with grade gets one active Enrollment.
- `User.current_enrollment` returns the active enrollment.
- Migration smoke: `migrate` runs clean (empty DB via test runner). Manual check: `migrate` on seeded SQLite copy + `makemigrations --check`.

Plus a minimal `apps/questions/tests/test_importers.py` for the extracted import function (valid JSON → N questions; bad JSON → error, no partial write).

## Execution / delegation

User asked for subagents. Workstreams (some parallel):
1. **WS-Academic** (B1.1–B1.5): new app, models, migration, admin, helper — single agent, internally sequential.
2. **WS-Admin** (U1 A1–A2): import action + importer refactor + permission-group seeding.
3. **WS-Tests**: pytest setup + tests — depends on WS-Academic + WS-Admin landing; run last / integrate.

WS-Academic and WS-Admin touch mostly disjoint files (academic/* + migrations vs questions admin/importers + seed command). `seed_initial_data` is touched by both (WS-Admin: perms; WS-Academic: academic seed) → sequence those two, or one agent owns the seed file. Decision: WS-Academic owns academic-seed function; WS-Admin owns perm-assignment; both edit `seed_initial_data` → run **sequentially** (Academic first, then Admin), not parallel, to avoid edit conflict. WS-Tests last.

Main thread integrates, runs `pytest` + `manage.py check` + `makemigrations --check`, updates `implementation-progress.md`.
